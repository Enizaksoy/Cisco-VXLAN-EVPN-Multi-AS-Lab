Overview

A complete BGP EVPN VXLAN fabric built on Cisco Nexus 9000v in Cisco Modeling Labs, using the Multi-AS (eBGP) design of RFC 7938: every leaf is its own autonomous system, the two spines share one AS and transport EVPN routes without a single VRF or VNI configured on them.

What makes this lab worth reading:
  • Every design rule was broken first and fixed second — the troubleshooting journal documents each fault with the exact CLI evidence.
  • DHCP relay behavior on a stretched L2VNI was analyzed at packet level with Wireshark, including a live single-point-of-failure test.
  • All configs and verification outputs were collected live from the running fabric.

Topology

Full-mesh leaf↔spine eBGP (8 point-to-point /30 links). The DHCP server (IOL) hangs off Leaf-1 on an 802.1Q trunk. Tenant hosts are lightweight Alpine Linux nodes spread across all four leaves.

Multi-AS eBGP Design

No IGP, no route reflectors: a single eBGP session per leaf-spine link carries both ipv4 unicast (underlay: loopback reachability) and l2vpn evpn (overlay).

Why spines share one AS

AS_PATH loop prevention becomes a free topology guard: a route trying to transit leaf→spine→leaf→spine is rejected by the second spine (own AS in path). Leaves can never become transit — valley-free routing enforced by numbering alone. Bonus: both spines present identical AS_PATHs, so ECMP works without multipath-relax.

Why each leaf gets its own AS

eBGP everywhere → no iBGP full-mesh/RR requirement; a leaf drops its own routes coming back (own AS in path); and AS_PATH instantly identifies the originating leaf during troubleshooting.

Spine: VRF-less EVPN Transit

The spine plays the role a VRF-less route reflector plays in MPLS L3VPN — l2vpn evpn is the exact analog of vpnv4:

MPLS L3VPNVXLAN EVPNPurpose
address-family vpnv4address-family l2vpn evpnVPN route transport
RDRDNLRI uniqueness
RT extcommunityRT extcommunityVRF import/export tagging
VPN labelVNIdata-plane tenant selector
PEleaf (VTEP)encap/decap, VRFs live here
RR (no VRFs)spine (no VRFs)blind route transport

EVPN routes sit in the spine's default-VRF BGP table as opaque NLRI. Three lines make it work — remove any one and the fabric fails in a different, silent way:

router bgp 65000
  address-family l2vpn evpn
    retain route-target all        ! (1) no VRFs → default RT filter would drop ALL EVPN routes
  neighbor 10.1.11.1
    address-family l2vpn evpn
      send-community extended      ! (2) RTs are extended communities
      route-map unchanged out      ! (3) eBGP rewrites NH; NH = remote VTEP = tunnel endpoint
route-map unchanged permit 10
  set ip next-hop unchanged

In the data plane the spine only routes the outer IP header (VTEP loopback → VTEP loopback); the VNI and inner packet are invisible to it — exactly like a P router switching labels.

Hard-won lesson: with leg 3 missing, show nve vni shows the L3VNI Down and every Type-5 stays invalid(EVI down) — visible in the local BGP table, advertised to no one. BGP never promises what the data plane can't deliver.

One host, two Type-2 routes

[2]:[0]:[0]:[48]:[5254.00f6.4040]:[0]:[0.0.0.0]/216       ← MAC-only  (producer: L2FWDER, label 30)
[2]:[0]:[0]:[48]:[5254.00f6.4040]:[32]:[172.16.30.1]/272  ← MAC-IP    (producer: HMM, labels 30 + 50001)

The MAC-IP flavor carries two labels (L2VNI + L3VNI) and two RTs — the signature of symmetric IRB — plus the originating leaf's Router MAC extcommunity, used as the inner Ethernet destination for routed VXLAN traffic.

Anycast Gateway & HMM

fabric forwarding anycast-gateway-mac 0000.1111.2222   ! global — identical fabric-wide
interface Vlan10
  vrf member Customer-A
  ip address 172.16.10.254/24                           ! identical IP on every leaf
  fabric forwarding mode anycast-gateway                ! per-SVI — the line everyone forgets

Without the per-SVI line, two leaves claim the shared IP with different system MACs:

%ARP-2-DUP_SRC_IP: arp Source address of packet received from 5269.d61a.1b08
on Vlan20(nve-peer1) is duplicate of local, 172.16.20.254

The source MAC being another leaf's router MAC (not a host MAC) pins this diagnosis. The second, quieter symptom: HMM (Host Mobility Manager) only converts ARP entries into MAC-IP Type-2 routes on anycast-mode SVIs — no anycast mode, no host routes, no ARP suppression.

The chain to remember: ARP → HMM → local-host-db → L2RIB → BGP Type-2 (+/32 host route). "MAC route exists but no MAC-IP" always means this chain is broken at the SVI.

DHCP Relay Deep Dive

Central DHCP server in Customer-A vlan30 territory; clients on stretched L2VNIs across four leaves. Full write-up: docs/dhcp-relay-deep-dive.md.

The relay is stateless — the state travels in the packet

Server's OFFER arrives addressed to the relay's own IP (giaddr), UDP 67:
1. BOOTREPLY to my own address       → server reply
2. read giaddr  → matches my Vlan40 SVI → egress interface known
3. read flags   → unicast or broadcast delivery
4. build frame: dst MAC = chaddr, dst IP = yiaddr   ← NO ARP! client can't answer ARP yet
5. transmit. Nothing to remember.

Verified live: the single-relay trap

With the relay on Leaf-2 only, hosts behind Leaf-4 still got leases — their broadcasts ride the stretched L2VNI (BUM/ingress-replication) to Leaf-2. It works as a side effect of flooding, not by design: when Leaf-2's nve went down, host-4 (own leaf perfectly healthy) lost DHCP.

"Working" ≠ "working by design". Configure the relay on every leaf carrying the VLAN, with a per-leaf unique ip dhcp relay source-interface loopbackN — the distributed anycast gateway makes the SVI IP (giaddr) non-unique, so server replies can otherwise land on a leaf with no UDP-67 listener and be silently dropped.

Control-Plane Security

Enabling relay moves DHCP from the flood path to the punt path: every Discover now costs CPU. Axiom: anything punted to the CPU is a DoS vector.

LayerMechanismEffect
edgestorm-control broadcastkills broadcast floods in the ASIC, before punt
edgeDHCP snooping (untrusted ports)drops rogue OFFER/ACK from host ports
edgeip dhcp snooping verify mac-addressframe src MAC must equal chaddr → defeats starvation tools
last lineCoPP copp profile strictper-class policers: a DHCP flood cannot starve BGP/ARP
show copp status
show policy-map interface control-plane | include dhcp|violate   ! rising 'violated' = attack signal

CoPP's elegance is isolation, not prevention — the bulkhead pattern: one flooded class cannot sink the ship.
